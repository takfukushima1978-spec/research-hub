01

TL;DR — このリポは何か

この一段落だけ読めば足りる

research-hub は、3つの経路 (Claude スケジュールタスク / ChatGPT GPTs / Gmail Import) で集めた記事を Supabase の research スキーマに集約し、静的 HTML ビューワーで読む個人リサーチ基盤。

リポジトリに居るのは「薄い Edge Function (insert-article / deep-research / articles) + 静的 index.html + マイグレーション + プロンプト」だけ。収集ロジック本体はリポ外（cron prompt と外部スクリプト）に置き、リポ側は「投入の受け口」と「品質ノルマ」「重複検知」だけ守る。

Deep Research は同じ Edge Function を request → list_pending → complete の三役で兼ねる非同期パイプライン。Web UI から 🔬 ボタンで投げ、ChatGPT/Claude 側で実行、結果書き戻し後に UI に自動表示される。

03

1日のフロー (3つの cron)

JST 基準で時系列に並べる。実線=cron 自動、点線=Tak / ChatGPT GPTs の手動。3つの scheduled task はすべて別の Claude Code の Remote Trigger。

自動処理 (cron / RPC) 手動 / 非同期 (Tak / GPT)

ポイント: insert-article が唯一の集約点。3つの cron も、ChatGPT GPTs も、Gmail Import も、最終的にここに集まる。品質ノルマ・重複検知をここに集中させれば全経路を一斉に守れる。TAK_NOTE: morning-email は DB を read してメール組み立てるだけで insert はしない (Step 1.5 確認)

05

設計判断の根拠

このノートの主役セクション。半年後に「なぜこう設計したんだっけ」と迷う可能性が高い 5 つの判断を残す。

各カードはデフォルトで折りたたまれている。タイトルをクリックして展開。

05-A · ADR候補

なぜ投入経路を 3 つに分けたか (単一フォームに統合しない)

「収集主体」が違うので、UI を統一しても運用は統一できない。受け口だけ統一する。

+
判断
Claude scheduled task / ChatGPT GPTs / Gmail Import の 3 経路をそれぞれ独立して残し、全経路を insert-article Edge Function に集約する。経路統合ではなく出口統合。
候補と却下理由
  • 単一 Web フォーム化: 「読んだ記事を貼って即投入」UI を作るだけで複数経路を消せそうだったが、Claude scheduled task は呼び出し主体が Claude 自身でフォーム入力に向かない。Gmail Import は受信を起点とした非対話処理が必要。TAK_NOTE: 検討時に却下した直接の理由を補強
  • Claude scheduled task に一本化: ChatGPT GPTs から Tak が直接「これも入れて」と渡せる経路を消したくなかった。両エージェントから書き込めることが価値。
  • Gmail Import を廃止: スマホで読んだ記事をスクショ的に取り込む経路として、メール下書き → 自動投入は手数が最も少ない。手動廃止の負担増が見合わない。
採用理由
  • 収集主体 (Claude / ChatGPT / Tak 本人 + Gmail) が違うので、UI 統合より出口 (insert-article) 統合のほうが現実的
  • 出口を 1 つにすれば、品質ノルマ・重複検知・embedding 生成を 1 箇所で守れる
  • 1 経路が壊れても他経路で記事が入り続ける冗長性が手に入る
  • TAK_NOTE: 実運用で「この経路で良かった」と思った瞬間があれば記録
前提条件
  • 3 経路すべてが INTERNAL_TOKEN を共有できる (現在: Edge Function Secrets で管理)
  • Gmail Import スクリプト本体は tak-work/リサーチ/auto-research/ 配下に git 管理外で存在 (40d609c で CLAUDE.md に明記済み)
  • TAK_NOTE: 他の前提があれば追記
再評価トリガー
  • 3 経路のうち 1 つが半年間使われなくなった (廃止候補)
  • 4 つめの経路 (例: Telegram からの投入、RSS リーダー連携) を本気で検討し始めた
  • ChatGPT GPTs が API 仕様変更で動かなくなった
  • TAK_NOTE: Gmail Import を別リポに切り出すか議論する場面
05-B · ADR候補

なぜ静的 index.html 単体ビューワーか (React / Next.js を採らない)

読み手は自分 1 人。フレームワークの学習・ビルド・デプロイの摩擦より、43KB の単体 HTML の方が圧倒的に速い。

+
判断
記事ビューワーは index.html 1 ファイル (CSS・JS インライン) で構成し、Supabase REST API を anon key で直叩きする。ビルドステップなし、GitHub Pages で配信。
候補と却下理由
  • Next.js (SSR): SEO・初期表示速度の旨味は個人利用では効かない。Vercel 環境変数管理・デプロイサイクルが「ちょっと UI 変えたい」摩擦を生む。
  • React SPA (CRA / Vite): ビルドステップが入る時点で、コミット → 反映の最短サイクルが伸びる。My URAWA LOG で React SPA を運用しているが、こちらは UI を頻繁に弄らない前提だった。
  • Supabase 公式ダッシュボード閲覧: 既読・クリップ・🔬 ボタンなど UI 操作が必要なので不可。
採用理由
  • 43KB 単体ファイル = 編集が Edit ツール 1 回で完結
  • GitHub Pages の git push → 数十秒で本番反映
  • Supabase REST 直叩きで「Edge Function を中継しない」分シンプル (RLS で守る)
  • 5dc694e / 594b638 / 351f584 のような細かい余白調整が毎週入る前提 → ビルドステップは負担
  • TAK_NOTE: 静的 HTML 路線への確信が揺らいだ瞬間 (例: 「やっぱり React にすればよかった」と思ったか) を残す
前提条件
  • Supabase 側の RLS で anon key が SELECT のみに制限されている
  • 記事数が「単一ページで全件描画してもブラウザが死なない」スケール (~数千)
  • TAK_NOTE: RLS の具体ポリシーを実装で確認する
再評価トリガー
  • 記事数が増えてページ描画が体感的に重くなった
  • UI 機能が複雑化し 1 ファイルで管理しきれない (フォーム多数・ルーティング必要)
  • 家族・他者にも閲覧させる必要が出た (認証が必要)
  • TAK_NOTE: ページネーション / 仮想スクロールの必要性が出たら
05-C · ADR候補

なぜ品質ノルマを 400 エラーで強制したか (warning にしない)

soft validate で連日「薄い記事 + 重複ネタ」が増えた事象を、出口で hard fail する設計に転換した。

+
判断
  • body_text ≥ 1500 / source_urls ≥ 3 / tag_names ≥ 4 を insert-article でハード強制
  • 未達は 400 を返し、details に不足項目を列挙して呼び出し側にリトライさせる
  • 手動投入のときだけ quality_override: true で逃がす
候補と却下理由
  • warning ログのみ (fail-open): 当初の設計。「とりあえず入れて後で見直す」運用が機能せず連日重複・薄っぺら問題が発生 → 却下。TAK_NOTE: 当時の Tak の判断「品質を犠牲にしてでも記事数を稼ぐ」設計が裏目に出た経緯を残す
  • クライアント側でバリデーション: 3 経路すべてに同じバリデーションを実装する負担が大きい。バージョン差で抜け穴が生じる。
  • insert は通して別ジョブで削除: 「入った後に消す」運用は事実上ゼロベース見直しを必要とし、続かない。
採用理由
  • 出口 (insert-article) は唯一の集約点なので、ここでの hard fail がもっとも効果範囲が広い
  • 設計パターン「fail-open / fail-closed を strict パラメータで両立」の strict 採用例 (本番=fail-closed、手動=override 抜け道)
  • 400 + details 形式で呼び出し側の Claude/ChatGPT が「何を直せば通るか」を自然言語で理解できる
  • 環境変数で閾値を上書きできるので、ローカル実験では緩められる
  • TAK_NOTE: 1500 / 3 / 4 の数字を決めた根拠 (実データの分布など) を補強
前提条件
  • auto-research-collect の prompt が「ノルマ前提で記事を構造化」できる粒度
  • 呼び出し側 (Claude / ChatGPT) が 400 を受けてリトライ・調整できる
  • 手動投入は人力で quality_override を付けられる (Gmail Import も)
再評価トリガー
  • ノルマ未達による rejection 率が高すぎて (例: 30% 以上) 投入数が日次目標を下回る
  • 逆に rejection 率がほぼゼロになり、ノルマが緩すぎる兆候が出た
  • 記事の質を判定する別軸 (例: 独自性スコア、出典の信頼度) が必要になった
  • TAK_NOTE: 半年運用しての rejection 率実績
05-D · ADR候補 + 用語あり

なぜ embedding 重複検知を opt-in にしたか

embedding 生成コスト負担を呼び出し側に持たせ、リポは「閾値を持つ判定ロジック」だけ提供する分離設計。

+
仕組みの確認

opt-in の意味: 呼び出し側が payload に embedding 配列 (1536 次元) を含めた場合だけ、insert-article が cosine 類似度を計算する。送らない既存フローは何も変わらない。

判定: cosine 類似度 ≥ 0.85 (デフォルト) の既存記事があれば 409 で重複拒否。embedding 生成は呼び出し側 (OpenAI text-embedding-3-small / Voyage AI voyage-3 等) の責務。TAK_NOTE: 0.85 を選んだ根拠を補強 (経験則 / 文献根拠)

判断
embedding は articles.embedding vector(1536) カラムと find_similar_articles_by_embedding RPC を用意。opt-in として、呼び出し側が embedding を送ってきた時だけ cosine 重複検知を発動する。
候補と却下理由
  • insert-article 内で embedding 生成 (mandatory): OpenAI API キーを Edge Function に持たせる必要があり、コスト管理・キー管理が複雑化。さらに「embedding 不要な手動投入」まで毎回 OpenAI を叩くのは無駄。
  • title+summary の文字列マッチで重複検知: 同じ意味で違う表現の記事を区別できない (例: 「OpenAI が GPT-5 を発表」と「サム・アルトマンが新モデルを公開」)。embedding でないと意味検知できない。
  • get_recent_article_digests による prompt 注入: すでに採用済み (auto-research-collect で重複テーマ除外)。これは「収集前の予防」、embedding は「投入時の最終ガード」と二段構え。
採用理由
  • embedding 生成コスト・API 鍵を呼び出し側に持たせ、リポ側は閾値判定だけ持つ責務分離
  • 既存の投入経路を一切壊さない (embedding を送らない経路は今まで通り)
  • 将来 embedding モデルを変えても、リポ側の RPC は無変更で済む (次元数が同じなら)
  • 409 重複拒否で呼び出し側がリトライ判断 (別アングルで再記事化 / 諦める) しやすい
  • TAK_NOTE: 採用後、実際に embedding を送ってきた経路はあるか / まだ opt-in を活用していないか
前提条件
  • pgvector 拡張が Supabase に有効化されている (urawa_log と同居プロジェクト、search_path 注意)
  • 呼び出し側が 1536 次元の embedding を生成できる (OpenAI text-embedding-3-small が標準)
  • cosine 閾値 0.85 が「ちょうど良い」前提 (経験則)TAK_NOTE: 実データで再評価する
再評価トリガー
  • opt-in を活用する経路が半年経っても出てこなかった (機能の存在意義の見直し)
  • 逆に embedding 重複検知の false positive / false negative が体感で気になり始めた (閾値調整)
  • 呼び出し側に embedding 生成負担を持たせる設計がコスト的に逆転した (insert-article で生成した方が安い)
  • Supabase が embedding 自動生成機能を出した
  • TAK_NOTE: 0.85 で誤って弾かれた記事の事例があれば
05-E · ADR候補 + パターン

なぜ Deep Research を request → complete の非同期分離にしたか

Web UI で 🔬 を押した瞬間に LLM を回さず、別エージェントが取りに来て返す。Web UI は LLM レイテンシから完全に独立。

+
判断
  • deep-research Edge Function を 3 つの action (request / list_pending / complete) で兼ねる
  • Web UI 🔬 → request で deep_research_requests に pending 行を作るだけ
  • ChatGPT/Claude が list_pending でポーリング → LLM 実行 → complete で結果書き戻し
  • Web UI は get_article_research で結果を取り、completed なら自動表示
候補と却下理由
  • Edge Function 内で同期的に Anthropic API を叩く: Deep Research は数分単位。Edge Function のタイムアウト・ブラウザの待ちで UX が破綻。TAK_NOTE: タイムアウト具体値 (Supabase Edge Function は 150s 程度) を確認
  • WebSocket / Server-Sent Events で進捗ストリーミング: 個人利用で同時実行 1 件、待つ必要すらない (通勤精読時に取れていればいい) ので過剰。
  • 外部キュー (Cloudflare Queues / SQS): Supabase テーブル 1 枚で同じことができる。インフラを増やすメリットなし。
採用理由
  • Web UI は request 即返却。LLM レイテンシから完全に独立
  • LLM 実行主体 (ChatGPT GPTs / Claude scheduled task) を入れ替え自由 (どちらが list_pending しても良い)
  • deep_research_requests テーブルに状態を持つので、再起動・障害に強い
  • 1 つの Edge Function に集約することで認証 (INTERNAL_TOKEN) を一括管理
  • TAK_NOTE: 「請求書 → 工事 → 検収」型の業務フローと同型構造。会計脳の比喩で残しておく
前提条件
  • ChatGPT GPTs または Claude scheduled task のどちらかが list_pending を定期実行
  • Web UI は「結果が後で表示される」UX を許容する (即時 LLM 応答を期待しない)
  • TAK_NOTE: list_pending を呼ぶ主体が ChatGPT 側か Claude 側か、現状の運用を確認
再評価トリガー
  • pending 滞留時間 (request → complete) が体感で長すぎると感じるようになった
  • 同時 Deep Research 件数が増え、優先度制御 (priority 1-3) が足りなくなった
  • Anthropic / OpenAI が「数分の LLM 呼び出しを安価にストリーム」できる API を出した
  • Supabase が Async Edge Function / Background Job 機能を出した
  • TAK_NOTE: list_pending のポーリング頻度の妥当性も再検討
06

既知の落とし穴

過去に踏んだ / 踏みそうな地雷。

  • Supabase Free tier のアイドルスリープ
    アイドル時にプロジェクトがスリープし、初回アクセスで 502 タイムアウト。auto-research-collect の最初の insert-article が落ちる可能性がある。リトライ前提で組むか、有料プランに上げるか。TAK_NOTE: 実際に踏んだ頻度を記録
  • ChatGPT GPTs の OpenAPI 仕様制約
    GPTs は同一ドメインで複数 Action を作れない (1 スキーマに統合必須)、oneOf 非対応。openapi-research-hub.json は単一 object スキーマで insertArticle / deepResearch を同居させてある。7ac3a24 で対応済み。
  • INTERNAL_TOKEN のヘッダ二系統
    X-Internal-Token と Authorization: Bearer の両方を受け付ける設計。ChatGPT GPTs は Bearer しか送れないので両対応にした経緯。デバッグ時にどちらが効いているか混乱しないこと。
  • Gmail Import スクリプトはこのリポジトリ外
    05_gmail_to_supabase.py は tak-work/リサーチ/auto-research/ 配下 (git 管理外)。端末乗り換え時に消失リスク。.claude/settings.local.json に allow 登録があるので「別プロジェクト混入」と誤判定されないよう CLAUDE.md (40d609c) に明記済み。TAK_NOTE: スクリプト本体をどこかに退避する仕組みを検討
  • Step 1.5 公式ニュース最優先 × 曜日ローテーション
    auto-research-collect は曜日別の軸ローテーションでネタを散らしつつ、Anthropic/OpenAI/Google DeepMind 等の公式リリースは別枠で必ず記事化する。重複記事が同日に増える可能性がある (技術詳細 / 競合比較 / 反対意見など別アングルで再記事化のルール)。記事数が増えすぎる兆候が出たら閾値見直し。
  • Supabase 同居プロジェクトの search_path
    Supabase プロジェクトは My URAWA LOG と同居 (memory: supabase-shared-with-urawa-log)。pgvector は urawa_log スキーマに居る。SECURITY DEFINER 関数の search_path に urawa_log を含めること。
  • scheduled task の cron は UTC 表記
    cron は UTC 基準 (例: 18:03 UTC) で書く。JST 換算 (03:03 JST) と混同しない。CLAUDE.md の表は UTC / JST 両方明記してある。
