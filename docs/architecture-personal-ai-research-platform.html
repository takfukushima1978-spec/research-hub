1. 全体アーキテクチャ概要

設計思想

本基盤は「収集 → 蓄積 → 検索 → 活用」の4レイヤーで構成されます。各レイヤーは疎結合に設計し、段階的に構築・拡張できる構造とします。

レイヤー間通信

  • 収集 → 蓄積: Supabase REST API（POST /rest/v1/articles）経由でJSON投入（Content-Profile: research ヘッダー付与）
  • 蓄積 → 検索: Supabase Edge Functionsでセマンティック検索APIを提供
  • 検索 → 活用: REST API / MCP Server経由で各ツールからアクセス

2.1 スキーマ作成 & 拡張の有効化

本リサーチプラットフォームのすべてのテーブル・関数は research スキーマに格納します。Supabaseを他プロジェクトと共用している場合でも、データが混在しません。

-- Supabase SQL Editorで実行

-- 1. 専用スキーマを作成
CREATE SCHEMA IF NOT EXISTS research;

-- 2. 拡張はpublicスキーマに作成（データベースレベル）
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- あいまい検索用
CREATE EXTENSION IF NOT EXISTS pgcrypto; -- SHA-256ハッシュ用

-- 3. search_pathにresearchスキーマを追加（セッション内でのアクセスを簡略化）
ALTER DATABASE postgres SET search_path TO public, research;

Edge Functionからのアクセス方法

Supabase REST APIのデフォルトは public スキーマです。research スキーマのテーブルにアクセスするには、リクエストヘッダーに Accept-Profile: research（GETの場合）または Content-Profile: research（POST/PATCH/DELETEの場合）を付与してください。
または、Supabaseダッシュボードの Settings > API > Exposed schemas に research を追加し、db_extra_search_path に設定する方法もあります。

2.9 セマンティック検索関数

-- コサイン類似度によるセマンティック検索
CREATE OR REPLACE FUNCTION research.search_articles(
  query_embedding vector(1536),
  match_threshold FLOAT DEFAULT 0.7,
  match_count INT DEFAULT 10,
  filter_category UUID DEFAULT NULL,
  filter_status TEXT DEFAULT 'published'
)
RETURNS TABLE (
  id UUID,
  title TEXT,
  summary TEXT,
  research_topic TEXT,
  published_at TIMESTAMPTZ,
  similarity FLOAT
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    a.id,
    a.title,
    a.summary,
    a.research_topic,
    a.published_at,
    1 - (e.embedding <=> query_embedding) AS similarity
  FROM research.article_embeddings e
  JOIN research.articles a ON a.id = e.article_id
  WHERE
    a.status = filter_status
    AND (filter_category IS NULL
         OR a.category_id = filter_category)
    AND 1 - (e.embedding <=> query_embedding) > match_threshold
  ORDER BY similarity DESC
  LIMIT match_count;
END;
$$;

3.2 body_html 保存ガイドライン

body_html にはCSSを含めず、<body>内のコンテンツHTMLのみを保存します。

  • 保存する: <h1>〜<p>タグ、テーブル、リスト等のコンテンツHTML
  • 保存しない: <style>タグ、インラインCSS、<head>セクション
  • body_markdown: Markdownに変換した軽量テキストも併せて保存（最も容量効率が良い）
  • フルHTML再構成: 表示時に css_templates のCSSと結合して生成

3.4 重複検知ロジック

3段階の重複チェックで、同じソースからの重複記事を排除します。

1
URL完全一致: research_sources.url_hashのSHA-256ハッシュで判定。既存URLならlast_seen_atとfetch_countを更新のみ。
2
タイトル類似度: pg_trgmのsimilarity()関数でタイトルの類似度を計算。閾値0.8以上なら重複と判定。
3
ベクトル類似度: 投入前にembeddingを生成し、既存記事とのコサイン類似度が0.95以上なら重複として投入をスキップ。 
-- 重複チェック用SQL関数
CREATE OR REPLACE FUNCTION research.check_duplicate(
  p_title TEXT,
  p_source_urls TEXT[],
  p_embedding vector(1536) DEFAULT NULL
)
RETURNS TABLE (
  is_duplicate BOOLEAN,
  duplicate_id UUID,
  match_type TEXT,
  similarity_score FLOAT
)
LANGUAGE plpgsql
AS $$
BEGIN
  -- 1. URL完全一致チェック
  RETURN QUERY
  SELECT true, a.id, 'url_match'::TEXT, 1.0::FLOAT
  FROM research.articles a
  WHERE a.source_urls && p_source_urls
  LIMIT 1;
  IF FOUND THEN RETURN; END IF;

  -- 2. タイトル類似度チェック
  RETURN QUERY
  SELECT true, a.id, 'title_similar'::TEXT,
         similarity(a.title, p_title)::FLOAT
  FROM research.articles a
  WHERE similarity(a.title, p_title) > 0.8
  ORDER BY similarity(a.title, p_title) DESC
  LIMIT 1;
  IF FOUND THEN RETURN; END IF;

  -- 3. ベクトル類似度チェック
  IF p_embedding IS NOT NULL THEN
    RETURN QUERY
    SELECT true, a.id, 'vector_similar'::TEXT,
           (1 - (e.embedding <=> p_embedding))::FLOAT
    FROM research.article_embeddings e
    JOIN research.articles a ON a.id = e.article_id
    WHERE 1 - (e.embedding <=> p_embedding) > 0.95
    ORDER BY e.embedding <=> p_embedding ASC
    LIMIT 1;
    IF FOUND THEN RETURN; END IF;
  END IF;

  -- 重複なし
  RETURN QUERY SELECT false, NULL::UUID,
    'none'::TEXT, 0.0::FLOAT;
END;
$$;

3.5 CSSテンプレート管理

HTML記事の body_html にはCSSを含めず、コンテンツHTMLのみを保存します。CSSテンプレートは専用テーブルで一元管理し、表示時にオンデマンドで結合します。

メリット

  • 1記事あたりのサイズを約9.2KB → 約5KB に削減（CSS 3.5KB分の重複排除）
  • CSSテンプレートの変更が全記事に一括反映
  • テーマ切り替え（ダークモード等）が容易

4. 検索・活用レイヤー設計

4.1 Supabase Edge Function: 統合検索API

セマンティック検索・キーワード検索・フィルタリングを統合する単一のエンドポイントを提供します。

// supabase/functions/search/index.ts
import { serve } from 'https://deno.land/std/http/server.ts'
import { createClient } from 'https://esm.sh/@supabase/supabase-js'

serve(async (req) => {
  const {
    query,          // 検索クエリ（自然言語）
    category,       // カテゴリフィルタ
    tags,           // タグフィルタ
    date_from,      // 日付範囲（開始）
    date_to,        // 日付範囲（終了）
    limit = 10,     // 取得件数
    search_type     // 'semantic' | 'keyword' | 'hybrid'
  } = await req.json()

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  )

  // 1. クエリのembedding生成
  const embeddingRes = await fetch(
    'https://api.openai.com/v1/embeddings', {
    method: 'POST',
    headers: {
      'Authorization':
        `Bearer ${Deno.env.get('OPENAI_API_KEY')}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      model: 'text-embedding-3-small',
      input: query
    })
  })
  const { data: [{ embedding }] } =
    await embeddingRes.json()

  // 2. セマンティック検索実行
  const { data: results } = await supabase
    .schema('research')
    .rpc('search_articles', {
      query_embedding: embedding,
      match_threshold: 0.7,
      match_count: limit,
      filter_category: category || null
    })

  return new Response(
    JSON.stringify({ results, count: results.length }),
    { headers: { 'Content-Type': 'application/json' } }
  )
})

5.4 データ鮮度管理

-- 90日以上前の記事を自動アーカイブ
CREATE OR REPLACE FUNCTION research.archive_old_articles()
RETURNS void
LANGUAGE plpgsql
AS $$
BEGIN
  UPDATE research.articles
  SET
    status = 'archived',
    archived_at = now()
  WHERE
    status = 'published'
    AND published_at < now() - INTERVAL '90 days'
    AND is_starred = false;  -- スター付きは保持
END;
$$;

-- Supabase Cron（pg_cron）で月次実行
SELECT cron.schedule(
  'archive-old-articles',
  '0 3 1 * *',  -- 毎月1日 03:00
  $$SELECT research.archive_old_articles()$$
);

アーカイブされた記事はembeddingを保持し、セマンティック検索では引き続きヒットしますが、デフォルトの一覧からは除外されます。完全削除は手動操作のみとします。

5.6 スキーマ分離の運用ガイドライン

本プラットフォームは research スキーマに完全分離されています。同一Supabaseプロジェクトで他のアプリケーション（public スキーマ）と安全に共存できます。

他プロジェクトとの共存ルール

  • テーブル名の衝突回避: 他プロジェクトは public スキーマを使用。本プラットフォームは research.* を使用するため、名前衝突は発生しません
  • RLS ポリシーの独立性: 各スキーマのRLSポリシーは独立して動作。他プロジェクトのポリシー変更は影響しません
  • バックアップ・リストア: pg_dump --schema=research で本プラットフォームのみの選択的バックアップが可能
  • 拡張機能の共有: pgvector, pg_trgm, pgcrypto は public スキーマにインストールされ、全スキーマから利用可能です

REST APIアクセス時のヘッダー設定
