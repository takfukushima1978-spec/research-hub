01

TL;DR — 30秒で思い出す

この一段落だけ読めば足りる

Research Hub v2は、毎朝Claude Codeのscheduled task daily-research がClaude Codeとリサーチ領域の最新情報を集め、Supabaseに構造化して保存し、ChatGPT Custom GPTとClaudeから両方検索可能にする個人用のリサーチ自動化基盤。

収集→保存→セマンティック検索→AIツール統合の四層構造。Markdown的な単純さよりも「半年後の自分が判断を再評価できる構造」を優先して設計してある。

04-A · ADR候補

なぜSupabaseか (vs Notion / Airtable / Firebase)

PostgresのRLSとSQL自由度を、ローコードのスピードで両立させたかった。

+
判断
記事メタデータ・タグ・既読状態の保存先としてSupabase (Postgres) を採用。
候補と却下理由
  • Notion: APIレート制限が厳しく、毎朝の一括書き込みに不向き。タグの階層構造もリレーショナルに扱いにくい。TAK_NOTE: 却下した時の決め手だった事例(具体的にレート制限で困った経験など)
  • Airtable: スキーマ柔軟性は高いが有料プランの行数制限とAPI料金がスケールしない。SQL検索ができない。TAK_NOTE: 試した経験 / 試さず却下したか
  • Firebase Firestore: NoSQLでクエリ表現が弱い。pgvectorのようなベクトル検索が組めない。TAK_NOTE: 検討したか / 候補にも入れなかったか
  • 自前Postgres + 自前API: 認証・RLS・REST APIの実装コストが大きい。1人プロジェクトでは過剰。
採用理由
  • PostgresのフルSQL + pgvectorでセマンティック検索まで一気に組める
  • RLSで「ChatGPT/Claudeに渡すanon keyから読み書きの粒度を制御」できる
  • freee-etl-engineで既にSupabase運用の知見があり、学習コストがゼロ
  • 無料枠で個人利用に十分
  • TAK_NOTE: ここに「決定打」を一文で。当時の自分が一番響いた理由
前提条件
  • 個人利用のスケール (記事数 ~数千、月間アクセス ~数百)
  • freee-etl-engineと同じSupabaseプロジェクトに同居している
  • TAK_NOTE: 他に前提があれば追記
再評価トリガー
  • 記事数が無料枠を超えそうになった
  • freee-etl-engineとSupabaseプロジェクトを分離する必要が出た
  • ChatGPT/Claude側でMCPベースのDB接続が標準化され、anon key経由が時代遅れになった
  • TAK_NOTE: 他にトリガーがあれば追記
04-B · ADR候補 + 用語解説あり

なぜtext-embedding-3-smallか

日本語・英語が混ざる記事タイトル/要約に対し、コストと精度のバランスが最良だった。

+
先に用語の確認 — embeddingとは何か

1. 一行で言うと

テキストを数百〜数千次元のベクトル(数字の列)に変換する仕組み。「意味が近いテキスト同士はベクトル空間で近くに来る」という性質を持つ。

2. なぜ必要か

従来のキーワード検索は「文字列が一致するか」しか見られない。「Claude Code」で検索しても、「Anthropicの開発支援ツール」と書かれた記事は引っかからない。embeddingを使うと意味で検索できる。「Claude Code」のベクトルと「Anthropicの開発支援ツール」のベクトルは近くにあるので、両方ヒットする。

3. text-embedding-3-smallとは

OpenAIが提供する埋め込みモデルの一つ。1536次元のベクトルを返す。同じシリーズに text-embedding-3-large (3072次元、高精度・高コスト) がある。smallはlargeの1/5程度のコストで、多くのタスクで実用十分な精度。

4. pgvectorとの関係

Supabase (Postgres) にはpgvectorという拡張があり、ベクトル列を保存して「コサイン類似度で近いものを引く」SQLが書ける。embeddingモデル(OpenAI)とベクトルDB(pgvector)は別レイヤー。TAK_NOTE: 半年後、ここの理解が深まっていたら書き直す

判断
埋め込みモデルとして OpenAI text-embedding-3-small を採用。
候補と却下理由
  • text-embedding-3-large: 精度は高いがコスト5倍。個人利用の記事数では精度差が体感できないと判断。
  • text-embedding-ada-002 (OpenAI旧世代): 3-smallより古く、性能・コストで劣る。
  • Cohere multilingual: 日本語性能は良いがOpenAIエコシステムから外れる。ChatGPT Custom GPTとの統合運用を考えると同じプロバイダで揃えた方が運用がシンプル。TAK_NOTE: 検討したか / 候補にも入れなかったか
  • Voyage AI / BGE系のローカルモデル: 自前ホスティングのコストが個人運用に合わない。
採用理由
  • 日本語と英語が混在する記事タイトル/要約で実用的な精度
  • 1536次元という「pgvectorで扱いやすい次元数」
  • OpenAI APIキー1本で完結 (ChatGPT Custom GPTと同じプロバイダ)
  • コスト: 1Mトークンあたり数セント程度で、個人運用ならほぼ無視できる
  • TAK_NOTE: 当時試した記事で実際に「意味検索が効いた」と実感した瞬間があれば記録
前提条件
  • OpenAI APIキーが安定して使える (環境変数管理済み)
  • 記事タイトル+要約のトークン数が短い (1記事あたり数百トークン以下)
  • TAK_NOTE: 他に前提があれば追記
再評価トリガー
  • OpenAIから後継モデルが出て、コスト/精度比が大きく変わった
  • Anthropic自身が embedding API を提供開始した (Claudeとの統合が深まる)
  • 多言語対応で精度差が体感できるレベルになった
  • TAK_NOTE: 他にトリガーがあれば追記
04-C · ADR候補 + セキュリティ設計

RLSポリシーの設計意図

anon keyを外部AIツールに渡す前提で、「読みは広く、書きは抹消不可カラムに限定」する。

+
判断
  • SELECT は anon ロールに対して全許可
  • UPDATE は is_read, is_clipped, read_at, clipped_at の4カラムに限定 (RPC関数経由)
  • INSERT/DELETE は service_role のみ (anonには許可しない)
  • service_roleキーは絶対に外部に出さない
候補と却下理由
  • anonに全UPDATE許可: 当初の設計はこれだったが、titleやsource_typeまで書き換え可能でリスクが高すぎ却下。
  • JWT認証 + ユーザ別RLS: 個人利用なのでJWT発行・管理のコストに見合わない。
  • service_roleキーをChatGPT GPTに渡す: 全権限が外部に出るので絶対NG。
採用理由
  • anon keyは「漏れても致命傷にならない」前提で設計
  • 「既読フラグ」「クリップフラグ」は書き換えられても損害が限定的
  • RPC関数経由にすることで「書き換えロジック」を集中させ、監査可能
  • ChatGPT Custom GPTにanon keyを設定して読み取り中心に使えるようになる
  • TAK_NOTE: 過去にChatGPT GPT用の設定をClaude Codeが生成した時、anon keyを直接埋め込んでしまった事故を経て厳格化した経緯。詳細を残す
前提条件
  • service_roleキーは絶対に外部に出ない (グローバルCLAUDE.mdにルール明記済み)
  • anon keyの漏洩は「想定リスク」として受容している (RLSが防御線)
  • add_manual_articleなどの書き込みRPCは SECURITY DEFINER + 入力バリデーション
再評価トリガー
  • 個人利用以外のユーザを想定する必要が出た (チーム化など)
  • anon keyが流出して実害が出た (この時点でRLS緩めるvs厳格化の判断)
  • SupabaseがRLSの新機能を出して書き換え粒度がさらに細かく制御できるようになった
  • TAK_NOTE: 他にトリガーがあれば追記
04-D · ADR候補 + 諸刃の剣

daily-researchをscheduled task化した諸刃の剣

毎日自動でリサーチが走る代償として、失敗時に「気づくのが遅れる」リスクを抱えた。

+
判断
Claude CodeのScheduled Task機能で daily-research を毎日 8:00 JST (cron: 0 23 * * * UTC) に自動起動。モデルは claude-sonnet-4-6。
候補と却下理由
  • 毎朝手動で起動: 続かない。「やる気がある日だけ」になり、リサーチが習慣化しない。
  • GitHub Actionsで自前cron: Claude Codeのコンテキスト (CLAUDE.md / navigator.md) を活用しにくい。Anthropic API直叩きになり、ツール統合の旨味が消える。
  • 常駐プロセス (PCに置く): PC稼働状態に依存。出張時に動かない。
採用理由
  • ホスト管理不要 (Claude Codeのクラウド環境で動く)
  • navigator.md・CLAUDE.mdをそのまま読ませられる
  • Max planの範囲内で運用可能
  • 失敗時のログがClaude Code側に残る
  • TAK_NOTE: 自動化に踏み切った時の心理的ハードル / 決め手
前提条件
  • Max plan継続
  • navigator.md / CLAUDE.md がリポジトリルートに正しく配置されている (起動ディレクトリのずれに弱い)
  • Anthropic API側の安定性 (500エラー時はそのまま落ちる)
  • TAK_NOTE: 他に前提があれば追記
再評価トリガー
  • scheduled task失敗が週1回以上発生するようになった (運用コストが手動越え)
  • Anthropic API料金体系が変わり、Max planだけではカバーできなくなった
  • Claude Code側のscheduled task仕様が変わった (廃止 / リソース制限など)
  • そもそも「毎日リサーチを読む」という習慣が変わった (週1で十分など)
  • TAK_NOTE: 半年運用して見えた現実のトリガー
05

既知の落とし穴

過去に踏んだ地雷。半年後の自分が同じ穴にハマらないように。

  • daily-researchの起動ディレクトリずれ
    scheduled task起動時のCWDがずれると navigator.md / CLAUDE.md がglobで見つからず "No files found" で止まる。リポジトリルート起動を確実にすること。TAK_NOTE: 解決策の最終形 (固定パス指定など) を記録
  • anon keyの平文埋め込み事故
    Claude Codeに「ChatGPT Custom GPTのSetup手順を作って」と頼むと、実際のanon key値を埋め込んだ手順を生成してしまったことがある。グローバルCLAUDE.mdに「秘密値はプレースホルダ置換必須」のルールを追加して再発防止済み。
  • 階層タグの固定UUID再構築リスク
    L1タグの初期データを手打ちUUID (a0000001-0000-...) で投入している。Supabaseプロジェクトを再構築・別環境に移す時、他プロジェクトとUUID衝突する可能性がある。TAK_NOTE: 移行が現実化したらgen_random_uuid()+CTE方式に書き換え
  • Anthropic API 500エラー時のサイレント失敗
    scheduled task実行中にAnthropic API側が一時的に500を返すと、タスクは落ちるが翌朝まで気づかない。status.claude.com の確認を運用に組み込むか、失敗通知を別チャネルに飛ばす設計が要検討。TAK_NOTE: 通知の仕組みを入れたら追記
